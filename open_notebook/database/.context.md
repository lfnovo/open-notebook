SurrealDB connection management, async repository base, and schema migration system. All database interaction in the application flows through this layer.

## Summary

- repository.py — async SurrealDB client wrapper with query helpers; base for all domain repository functions
- async_migrate.py — `AsyncMigrationManager`: discovers and applies `.surql` migration files on API startup
- migrate.py — synchronous migration runner (CLI/scripts use)
- migrations/ — numbered `.surql` migration files defining schema evolution (e.g. `001_init.surql`, `002_embeddings.surql`)

## Constraints

- MUST number migration files sequentially; the migration manager applies them in order and records the last applied version.
- MUST NOT modify already-applied migration files; create a new numbered file instead.

## Guidance

- SHOULD write idempotent migrations (use `DEFINE ... IF NOT EXISTS` / `DEFINE ... OVERWRITE` where supported).
